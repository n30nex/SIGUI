# Known Limitations

As of the 2026-06-30 D1L host-simulator checkpoint:

- Manual visual confirmation of display bars and touch target movement is still pending; the serial `display test` and `touch test` commands return OK on hardware.
- MeshCore Public group text TX/RX and signed advert TX/RX are implemented and validated through the serial console, firmware-local contacts can promote heard nodes by fingerprint, full advertised public keys are retained for DM targeting and have passed reboot persistence on `COM7`, contact favorite/mute flags persist across reboot, Public unread/read state persists across reboot, per-thread DM read cursors are implemented, firmware-local routes are learned from Public/advert path metadata, route detail diagnostics and a first route detail sheet are implemented, and serial DM flood TX plus the bounded DM store are validated. DM ACK/PATH receive parsing, direct-route TX selection, contact-row DM compose, first contact detail actions, a bounded Public History/search sheet, and a bounded scrollable DM thread/detail history sheet are implemented, and targeted outbound hardware DM receive by the local COM11 Meshcorebot has passed. Manual touch review, controlled ACK/PATH RF proof, inbound DM proof, controlled inbound DM unread proof, and direct-route RF proof are still pending.
- Settings are implemented as an NVS-backed firmware model and have passed reboot persistence smoke on real D1L flash. The persisted radio profile is now exposed through `settings get`, serial radio setters, and a staged touch Radio Settings sheet; the radio editor slice has passed hardware flash, standard smoke, targeted serial profile proof, and active Public `test` RF regression on `COM7`. Manual touch review is still pending.
- First-boot onboarding now persists a setup-complete flag, collects the node name, confirms Canada/USA defaults, keeps Wi-Fi/BLE/observer disabled, and generates/retains the local MeshCore identity. Optional-radio choices in onboarding remain limited until live Wi-Fi/BLE runtime support is enabled.
- `identity status` now generates and reports an NVS-retained Ed25519 local identity fingerprint. This is a minimal firmware identity model, not the final full MeshCore C++ store integration.
- MeshCore TX timestamps are retained monotonically in NVS to avoid duplicate filtering, but a real RTC/network time source is not wired yet.
- MeshCore 3-byte companion framing is implemented and host-tested, but the live binary bridge to MeshCore protocol frames is not enabled in Phase 1.
- Packet log storage keeps a 32-row RAM ring and persists the newest 8 rows through the retained blob-store backend. It can use the RP2040 SD file protocol when the bridge reports a ready card, file operations, matching path/line/chunk limits, and atomic rename; NVS remains mirrored as fallback. Packet rows carry a bounded raw hex preview and expose serial filter/search/raw commands plus touch Packet-tab filters/search; larger retention, export, and deeper protocol decode are still pending.
- Optional SD-card data storage is now an important roadmap item with ESP32-side RP2040 status/setup plumbing and a CI-buildable RP2040 SD bridge target. Current ESP32 firmware reports `storage status`, including file-operation capability gates and additive bridge probe fields, probes the RP2040 with `DESKOS_SD_STATUS`, exposes non-formatting `storage diag`, exposes `storage setup` plus a touch Storage Setup sheet, can switch retained Public/DM message history, route history, and packet history to SD through `DESKOS_SD_FILE v=1` when the bridge is ready, and does not run boot-time or incidental formatting. The serial-only `storage map-policy` command and first touch Map tab expose the offline cache path contract, cache readiness, manual center state, and `download_supported=false` state without Public RF or formatting; `map center set <lat> <lon>` persists an optional manual center in settings. The serial-only `storage filecanary` command can prove temp write, read-back, `rename replace=1`, stat, final read, delete, and cleanup after the RP2040 bridge reports ready file operations. The serial-only `storage export-canary <token>`, `storage export-diagnostics <token>`, `storage export-data <token>`, and `storage map-tile-canary <token>` commands can prove diagnostic export, sampled user-data export, and map-tile cache commit paths through temp write/read and atomic rename without Public RF or formatting; the diagnostic bundle reports whether the map-tile cache backend is ready but does not bundle actual tiles, and the data export excludes private identity material. The RP2040 bridge now uses a raw SdFat card probe and SPI1-aware formatter so confirmed formatting is available only after a status reply explicitly reports a present card, `format_supported=true`, and setup-required state. GitHub Actions run `28490556715` produced verified ESP32/RP2040 artifacts and the ESP32 image was flashed on COM12; COM12 smoke passes, but the RP2040 UF2 has not been copied because no UF2/BOOTSEL drive is mounted. Current preflight reports `sd_card_not_present_diag_pending`, `rp2040_protocol_supported=true`, `rp2040_diag_supported=false`, `sd.state=no_card`, and `next_action="put_rp2040_in_uf2_bootloader"`, with no format and no Public RF. Settings, identity, contacts, and read-state remain onboard/fallback-backed or pending until later migrations are implemented and hardware-proven. Live network map tile downloads, touch location entry, GPS/location-source integration, and browser/geolocation integration remain pending. Long validation sessions can still hit NVS pressure until the RP2040 bridge is flashed and SD-backed history is proven on hardware. The intended production behavior remains boot-time SD detection/format validation, user-confirmed formatting for unusable cards, normal onboard-storage fallback when no usable card is found, and SD-backed larger stores for message history, packet/route history, exports, and future map tiles.
- Phase 6 mesh visibility now exposes read-only `signal`, `roomservers`, `repeaters`, and `routes trace <fingerprint>` diagnostics plus Home/Packet-tab summary cards, and the serial/summary slice is built, flashed, smoke-tested, reboot-checked, and Public `test` RF-regression tested on `COM7`. A first touch Mesh Roles browser sheet and read-only contact Route Trace sheet are also implemented; manual physical touch confirmation of opening/scrolling/closing them is still pending. Active RF ping/trace helpers and richer filters are still pending.
- Contact export now emits MeshCore-compatible `meshcore://contact/add?...` payloads over serial and adds a Contact Detail `Export` action with LVGL QR rendering enabled in the default build. It requires a promoted contact with a retained 64-hex public key. The QR-enabled build passed COM7 flash, standard smoke, targeted `contacts export <fingerprint>` proof, and active Public `test` RF regression; manual touch open/close review and scan/import proof with a MeshCore client are still pending.
- The Phase 3/4/5/6 touch shell is implemented with first-boot onboarding, Home, Public `test` action, first Public free-text composer, bounded Public History/search, first contact-row DM composer, first contact detail sheet with DM/favorite/mute/Trace/Edit actions, a contact Edit sheet for alias rename and promoted-contact Forget, bounded DM thread/detail retained history with `Reply` and per-thread `Read`, persisted recent Public/DM preview rows, Public/DM unread counters, newest contact/heard-node rows, an offline Map tab with SD tile-cache policy/readiness, retained route evidence, and serial-configured manual center display, Home/Packet-tab signal and mesh-role summary cards, first Mesh Roles browser sheet, first contact Route Trace sheet, Packet-tab route rows with a first route detail sheet, Packet-tab packet rows with filters/search/raw-hex detail, Settings companion status, a staged Radio Settings sheet, Storage Setup sheet, advert sheet, toasts, and lock overlay, but active route ping/trace helpers, touch map center entry, GPS/location-source integration, live map tile downloads, BLE companion transport, live Wi-Fi scan/connect, and OTA are not implemented yet.
- `tools/ui_simulator.py` now provides deterministic 480x480 host screenshots, schema-v2 required-label/text-overflow checks, a touch-target map with expanded 44x44 boxes, expected flow coverage for onboarding, lock/unlock, Public compose/history/search, DM thread read/reply, contact detail/edit/export/trace, Map policy/manual center, packet search/detail/routes, Mesh Roles, Radio, Storage, and Advert, plus RF/destructive/no-format action flags. It also includes storage-state scenarios, a `manual-location` scenario, and a `large-mesh` stress scenario for oversized node/message stores. This is a fast CI regression check, not a replacement for the pending manual physical-screen review of touch behavior, LCD color/brightness, animation smoothness, touch-controller accuracy, or genuinely large live RF mesh behavior.
- The Public composer, bounded Public History/Search sheets, contact-row DM composer, contact detail sheet, contact Route Trace sheet, contact export sheet, Radio Settings sheet, bounded retained-history DM thread sheet, per-thread DM read action, Messages unread/read action, route detail sheet, and packet detail sheet have passed host contract/simulator tests. Public History/Search has also passed hardware flash, standard smoke, and a targeted COM11 direct-message regression without Public-channel RF; `scripts/probe_d1l_dm.py` now makes that COM11 DM-only regression repeatable. The contact export, radio settings, per-thread DM read, and retained-history DM thread additions have also passed hardware flash, standard smoke, targeted serial proof, and active Public `test` RF regression on `COM7`. Manual physical touch entry on the composer keyboard and public-history/thread/reply/history/contact/export/radio/read/route-trace/route-detail/packet-detail surfaces is still pending.
- The SX1262 Phase 1 probe intentionally avoids TCXO/DIO3 control. TCXO remains `NONE` by default.
- Wi-Fi and BLE are managed by a Phase 5 status/persistence foundation. This build reports Wi-Fi support compiled in but not started by DeskOS, and BLE build-disabled, so `wifi scan` reports why scanning did not start instead of listing networks.
- Phase 7 diagnostics now expose a small reset/crash ring and richer health telemetry. A repeatable soak runner now samples health, mesh, signal, unread, packet, command retries, and crashlog state for idle/listening or active Public `test` windows, and a retry-aware 1-hour active Public `test` hardware soak passed on `COM7`; the full 12-hour idle/listening soak is still pending.
