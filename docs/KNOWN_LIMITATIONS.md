No supported-SDK waiver or hardware qualification is implied by the standalone green build.

- Settings are implemented as an NVS-backed firmware model and have passed reboot persistence smoke on real D1L flash. The persisted radio profile is exposed through `settings get`, serial radio setters, and a touch Radio Settings sheet; the user-facing More page groups tools, connections, storage/maps, device, support, and advanced functions behind disclosure rows instead of a flat settings grid. - First-boot onboarding now persists a setup-complete flag, collects the node name, confirms Canada/USA defaults, keeps Wi-Fi/BLE/observer disabled, and generates/retains the local MeshCore identity. Optional-radio choices in onboarding remain limited until live Wi-Fi/BLE runtime support is enabled.
- `identity status` now generates and reports an NVS-retained Ed25519 local identity fingerprint. This is a minimal firmware identity model, not the final full MeshCore C++ store integration.
- MeshCore TX timestamps are retained monotonically in NVS to avoid duplicate filtering, but a real RTC/network time source is not wired yet. The current production patch keeps DM/Public TX from failing under onboard NVS capacity pressure by falling back to a monotonic RAM timestamp if only the timestamp persistence write fails; that fallback must be hardware-retested from the next Actions-built firmware package.
- The first #69 durability slice removes synchronous full-route rewrites and storage reinitialization from hot RX observations. A dedicated persistence worker gives the first material observation a full 5-second coalescing floor, flushes nonmaterial-only dirty state by 30 seconds, and handles explicit migration/reconciliation immediately; failed forced flushes now honor the same five-second retry floor instead of hammering storage. Controlled reboot force-flushes or cancels, and `routes` exposes commit/coalesced/failure/dirty/reconcile truth. SD retains the newest 16-route primary snapshot while NVS mirrors only the newest 4 routes. Retained Public/DM/route/packet mirrors use a dedicated 124 KiB `d1l_retained` NVS partition, marker format v2 with two slots in the separate 4 KiB metadata sector, a committed anchor inside the dedicated NVS, and a completion sentinel in default NVS rather than competing with settings, identity, Wi-Fi, contacts, and nodes in the 24 KiB default partition. Upgrade reads prefer the dedicated partition, copy an old default-NVS retained key before erasing only that key, preserve and fail closed on divergent downgrade/re-upgrade copies, and commit the completion sentinel only after migration succeeds. Marker- or sentinel-owned recovery never explicitly erases `d1l_retained`, sentinel-only marker repair requires the already-committed anchor, and the sole blank owned-state resume is marker 1 valid, marker 2 erased, and no completion sentinel. Published pre-anchor v1 markers remain valid ownership: firmware initializes and commits the anchor without a data erase, migrates, commits the sentinel, then rewrites both marker slots as v2. `storage status.retained_nvs` must report `marker_ready=true`, `markers_complete=true`, `anchor_ready=true`, `sentinel_ready=true`, and `external_init_required=false` before the backend is release-ready. A late or transiently unreadable SD primary is read and merged by epoch/total before any overwrite, including chronologically replayed live overlays; persistent SD read failure still commits an independently snapshotted compact NVS fallback while keeping SD reconcile and controlled reboot fail-closed. A synchronized monotonic backend generation forces this read/merge again after every observed remove/reinsert cycle even when the worker never samples the intermediate disabled level. Every retained SD writer captures that generation and rechecks it immediately before replace-rename; a detected mid-write media edge leaves the primary untouched and does not issue a cleanup delete against the replacement card. Persisted route envelopes reject unterminated/non-printable strings, invalid sequence metadata, duplicate sequences, and impossible path fields before exposing rows. A same-epoch newer legacy snapshot wins for rollback freshness while a newer v2 epoch blocks stale legacy resurrection. Volatile UI route canaries use a separate preview slot and do not consume durable sequence/count/revision/capacity, and a failed `routes clear` blocks every periodic/forced persistence attempt until an explicit clear succeeds. The dedicated partition still requires exact Actions/COM12 upgrade and reboot proof and does not close #69: Public/DM/packet do not yet perform route-equivalent read/merge before a late-ready or replacement SD write, and their successful-SD/failed-NVS mirror case has no retryable dirty state. Checksummed-schema, other-store coalescing, power-loss, factory-reset, and truthful-time work also remain open.
- ESP-IDF's `nvs_flash_init_partition` is not a read-only discriminator: initialization may erase or activate a corrupt page. Firmware therefore performs zero retained-region erases and does not call NVS initialization on ambiguous nonblank bytes with neither marker nor sentinel ownership; it reports `external_init_required=true`. A separately audited installer/hardware step must verify the supported predecessor layout before erasing only `d1l_retained`; firmware then verifies blank first use and establishes marker 1, NVS, anchor, marker 2, migration, and sentinel in order. The known failed pre-anchor recovery cannot be authorized by a similar-looking live status: the installer also requires the hash-pinned incident manifest, exact historical receipt hashes, exact table artifact, live ESP32-S3 MAC, the audited prior MeshCore identity, and exact full raw-region SHA, then records durable pre-erase/erase/verify stages without retaining a raw backup. Simultaneous loss of both markers and the default sentinel, including an anchor-only NVS, stays fail-closed and preserved until that external decision; it is not an automatic deletion case.
- NVS initialization is fail-closed and non-destructive: DeskOS no longer erases the whole NVS partition automatically for `ESP_ERR_NVS_NO_FREE_PAGES` or an unsupported stored NVS version. The boot event and `health` expose a per-boot nonce plus `nvs_ready`/`nvs_error`; persisted settings remain untouched for explicit recovery. In this degraded mode NVS-backed features may be unavailable until the underlying capacity/version problem is resolved, and exact hardware recovery UX remains open under #69.
- MeshCore 3-byte companion framing is implemented and host-tested, but the live binary bridge to MeshCore protocol frames is not enabled in Phase 1.
- Packet log storage keeps a 128-row active RAM ring and persists the newest 8 rows through the compact NVS fallback. When the RP2040 SD file protocol is ready, packet history uses a split retained-store write path: SD receives the primary snapshot through coalesced flushes while NVS keeps the newest fallback rows, and a validated 4096-record segmented SD journal backs 100-row Packet tab pages with Load Older/Newer. Packet rows carry a bounded raw hex preview and expose serial filter/search/raw commands plus a touch terminal-style Packet tab with RX/TX/fail/error coloring, pause/resume, filter/search, route evidence, and normal/advanced packet detail; full hardware proof of 24h SD scrollback and deeper protocol decode are still pending.

D1L has no onboard GPS; Map center is manual and DeskOS makes no local-GPS claim.
