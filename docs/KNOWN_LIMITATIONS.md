# Known Limitations

As of the 2026-07-10 D1L COM12/COM16 production-roadmap checkpoint:

- ESP-IDF v5.1.x is end of life and is no longer the selected production target. Standalone migration commit `39a043c` uses version-pinned `espressif/idf:v5.5.4`, commits the exact Actions-generated `dependencies.lock`, applies the tracked Seeed BSP compatibility patch, and passes host, firmware, package, checksum, release, and effective-config checks. That evidence does not yet qualify the combined Map/Wi-Fi candidate: exact COM12 must still report `"idf":"v5.5.4"` and pass board/display/touch/Wi-Fi/RF/RP2040-SD/Map/health/reboot/power-cycle smoke with refreshed commit-matched release evidence. No supported-SDK waiver or hardware qualification is implied by the standalone green build.
- Manual visual confirmation of display bars and touch target movement is still pending; the serial `display test` and `touch test` commands return OK on hardware. The COM12 hardware pixel-capture path reconstructed the current PR #56 hierarchy from `51258ba` / Actions `29060900359` with matching firmware/host CRC `E72745BA`, a passing simulator diff, `public_rf_tx=false`, and `formats_sd=false`; a three-round all-tab health probe also passed cleanly. PR #51, PR #40, and PR #33 captures remain history for superseded layouts. Physical photos/manual review and final-release-SHA evidence are still pending.
- MeshCore Public group text TX/RX and signed advert TX/RX are implemented and validated through the serial console, firmware-local contacts can promote heard nodes by fingerprint, full advertised public keys are retained for DM targeting and have passed reboot persistence on `COM7`, contact favorite/mute flags persist across reboot, Public unread/read state persists across reboot, per-thread DM read cursors are implemented, firmware-local routes are learned from Public/advert path metadata, route detail diagnostics and a first route detail sheet are implemented, and serial DM flood TX plus the bounded DM store are validated. DM ACK/PATH receive parsing, direct-route TX selection, contact-row DM compose, first contact detail actions, a bounded Public History/search sheet, and a bounded scrollable DM thread/detail history sheet are implemented, and targeted outbound hardware DM receive by the local COM11 Meshcorebot has passed. `scripts/rf_full_acceptance_d1l.py` now provides a single COM12-side artifact path for identity, Meshcorebot status, outbound DM, controlled inbound DM, ACK/PATH, direct-route, health, and no-Public-command proof. Manual touch review, a passing full RF acceptance artifact, controlled inbound DM unread proof, and physical DM workflow review are still pending.
- Settings are implemented as an NVS-backed firmware model and have passed reboot persistence smoke on real D1L flash. The persisted radio profile is exposed through `settings get`, serial radio setters, and a touch Radio Settings sheet; the user-facing More page groups tools, connections, storage/maps, device, support, and advanced functions behind disclosure rows instead of a flat settings grid. PR #56 proves the collapsed hierarchy and all-tab navigation on COM12; expanded category touch behavior and the remaining system-page redesigns still need physical touch review.
- First-boot onboarding now persists a setup-complete flag, collects the node name, confirms Canada/USA defaults, keeps Wi-Fi/BLE/observer disabled, and generates/retains the local MeshCore identity. Optional-radio choices in onboarding remain limited until live Wi-Fi/BLE runtime support is enabled.
- `identity status` now generates and reports an NVS-retained Ed25519 local identity fingerprint. This is a minimal firmware identity model, not the final full MeshCore C++ store integration.
- MeshCore TX timestamps are retained monotonically in NVS to avoid duplicate filtering, but a real RTC/network time source is not wired yet. The current production patch keeps DM/Public TX from failing under onboard NVS capacity pressure by falling back to a monotonic RAM timestamp if only the timestamp persistence write fails; that fallback must be hardware-retested from the next Actions-built firmware package.
- MeshCore 3-byte companion framing is implemented and host-tested, but the live binary bridge to MeshCore protocol frames is not enabled in Phase 1.
- Packet log storage keeps a 128-row active RAM ring and persists the newest 8 rows through the compact NVS fallback. When the RP2040 SD file protocol is ready, packet history uses a split retained-store write path: SD receives the primary snapshot through coalesced flushes while NVS keeps the newest fallback rows, and a validated 4096-record segmented SD journal backs 100-row Packet tab pages with Load Older/Newer. Packet rows carry a bounded raw hex preview and expose serial filter/search/raw commands plus a touch terminal-style Packet tab with RX/TX/fail/error coloring, pause/resume, filter/search, route evidence, and normal/advanced packet detail; full hardware proof of 24h SD scrollback and deeper protocol decode are still pending.
- Optional SD-card data storage is FAT32-only and non-destructive. Merged PR #61 / source `4d9f384` proved the read-only Storage/Card/Data hierarchy on exact Actions-built COM12 captures with CRCs `9878ADFB`, `77726394`, and `42619B0E`, passing simulator diffs, moved-and-restored scrolling, and a clean three-round probe without Public RF or formatting. Manual physical touch/photos remain open, and this UI proof is not new SD matrix evidence. Current serial commands and the RP2040 bridge remain non-formatting, and users must provide FAT32 cards prepared on a computer. Existing core file/history/export/map evidence remains valid, while the remaining physical SD matrix, tile rendering, GPS/location-source integration, and nearby GPS pins remain pending.
- Older guarded-format hardware artifacts from July 1-2, 2026 are now obsolete for release readiness because the public policy forbids on-device formatting. Fresh SD acceptance evidence must use a FAT32 card prepared on a computer, prove auto-creation of the DeskOS directory structure, prove no-card NVS fallback, and prove non-FAT32/unmountable cards remain non-destructive with clear repair guidance.
- Current COM12/COM16 evidence from `1a29876` / GitHub Actions run `28714355561` proves the core SD path on the installed FAT32 card without Public RF or formatting: the official Seeed SD smoke passes on COM16, COM12 smoke reports `READY_SD`, raw RP2040 diagnostics complete, the file canary passes, correct/missing/existing-data boot scenarios pass, RP2040-unavailable fallback is captured after official smoke, retained Public/DM/route/packet history survives reboot, reboot/remount passes, the synthetic map-tile canary passes, and export canary/diagnostic export/sampled data export all pass. Public release is still blocked on the remaining physical SD evidence matrix: actual no-card and unformatted/non-FAT32 scenarios, FAT32-only rejection behavior for unusable media, <=32GB multi-card proof, no-format guidance evidence tied to unusable media, and power/electrical evidence.
- Phase 6 mesh visibility exposes `signal`, `roomservers`, `repeaters`, routes, and Home/Packet summaries. PR #60 / source `0b138be` proved the full-height read-only Mesh Roles hierarchy on exact Actions-built COM12 captures with CRCs `63DE54FB`, `FD538D71`, and `5C41EE08`, passing diffs and a clean three-round probe without Public RF or formatting. Manual physical Back/scroll review remains pending.
- Contact export emits MeshCore-compatible `meshcore://contact/add?...` payloads over serial with LVGL QR rendering enabled in the default build and requires a promoted contact with a retained 64-hex public key. Contact Detail places Message and Contact options at the root; Route, Export, Rename, favorite/mute state, and confirmed removal live under Contact Options. Merged PR #59 / `d24552e` / Actions `29064260772` proved the Contact Detail, Contact Options, and confirmation-only Forget pages on COM12 with exact CRC matches and passing simulator diffs, without invoking deletion, Public RF, or SD formatting. Physical touch review and scan/import proof with a MeshCore client remain pending.
- The touch shell implements Home/Messages/Network/Map/More plus full-height nested messaging, contact, Mesh Roles, and Storage pages. PR #56 proves Home, PR #58 proves DM Thread, PR #59 proves contacts, PR #60 proves Mesh Roles, PR #61 proves Storage, and PR #35 proves the retained keyboard callers. The revised built-in-source Map starts at zoom 10, permits zooms 8 through 14, and exposes one-finger pan plus direct `-`, `+`, and `Center` controls while retaining `Map -> Map options -> Set location or Cache status`. Pan/zoom are session view state rather than saved preferences. The revised controls and completed exact-view Home-to-Map retained-frame reuse still need commit-matched Actions, combined-COM12 pixels, and live request/render/cancel/cache/heap proof. Background fetch, multi-zoom prefetch, off-screen arbitrary-coordinate batches, and area download are intentionally unsupported.
- `tools/ui_simulator.py` provides deterministic 480x480 host screenshots, schema-v2 required-label/text-overflow checks, 44x44 touch-target checks, and expected flows. Map scenarios cover no location, Wi-Fi off, Wi-Fi connecting, live current-view, and cached revisit states; they record the zoom-10 default, 8-through-14 range, one-finger pan, direct controls, a 9-tile visible limit, one zoom per visible generation, completed same-view frame reuse, no background/prefetch/area download, and visible `(c) OpenStreetMap contributors`. This is a fast CI regression check, not a replacement for physical control/tile/render/network/memory proof.
- A UI refresh fix has been added for the reported split-page/live-packet corruption: RF-mutated packet/message/DM/node/route/contact stores and mesh-inspector scratch buffers serialize reads and writes before UI snapshots, touch callbacks request coalesced content refreshes, and live data generation changes queue redraws for data-dependent tabs. The revised Map acknowledges unrelated live counters and transient SD busy/readiness changes without tearing down its viewport, drag, or unfinished tile batch; only changed saved-location/default-zoom inputs request its full redraw. Reopen Map after inserting or restoring an SD card to retry. `59610ab` / Actions `28723265336` passes the earlier longer targeted corruption probe, PR #56 / Actions `29060900359` passes the current hierarchy Home capture plus three-round all-tab health proof, and PR #35 / Actions `28727064923` passes the all-target keyboard capture. The revised Map guard still needs commit-matched Actions and COM12 proof. Final release also needs physical photos/manual review, full RF acceptance, remaining SD matrix evidence, remaining #16/#6 UI ownership evidence, and the 12-hour soak.
- Messaging, contact, packet, radio, hierarchical Mesh Roles, and Storage pages have host coverage; PR #58 proves DM Thread, PR #59 proves contacts, PR #60 proves Mesh Roles, and PR #61 proves Storage on COM12. The current Map hierarchy has standalone Actions coverage, but its combined-candidate pixels and physical live-flow proof remain open; manual physical review remains open across these surfaces.
- The SX1262 Phase 1 probe intentionally avoids TCXO/DIO3 control. TCXO remains `NONE` by default.
- Wi-Fi and BLE are managed by an offline-first companion-radio policy. Wi-Fi remains off by default, but the Phase D station path can be explicitly enabled for bounded `wifi scan`, saved-profile `wifi connect`, and touch Wi-Fi Setup use without printing saved passwords. BLE companion transport remains build-disabled/pending. Standalone `de79c9f` passes exact-Actions-artifact Wi-Fi hardware proof; the full enable/connect/reconnect/reboot/low-memory path must be repeated on the combined ESP-IDF v5.5.4 artifact.
- Phase 7 diagnostics now expose a small reset/crash ring and richer health telemetry. A repeatable soak runner now samples health, mesh, signal, unread, packet, command retries, and crashlog state for idle/listening or active Public `test` windows, and a retry-aware 1-hour active Public `test` hardware soak passed on `COM7`; the full 12-hour idle/listening soak is still pending.
- `scripts/release_gate_audit_d1l.py` provides a machine-readable, commit-metadata-strict release audit and now includes a P0 `supported_sdk_baseline` gate for exact `espressif/idf:v5.5.4` workflow selection and a committed lock that records IDF 5.5.4. That gate does not prove the lock's Actions provenance and does not replace the issue #63 clean-build, COM12 `version.idf`, behavioral, or final evidence requirements. Targeted corruption/scroll evidence is proven on `59610ab` / Actions `28723265336`, all-target keyboard capture is proven on PR #35 / Actions `28727064923`, and current hierarchy pixel/navigation proof is proven on PR #56 / Actions `29060900359`. These issue-sized artifacts do not replace the final frozen release commit's matching evidence. Release remains fail-closed until issue #63 qualification, RF/DM proof, remaining physical SD matrix proof, 12-hour idle/listening soak, manual physical UI/photos, remaining #16/#6 architecture evidence, and the final commit-matched sweep pass.
