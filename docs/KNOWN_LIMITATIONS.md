# Known Limitations

As of the 2026-07-04 D1L COM12/COM16 production-roadmap checkpoint:

- Manual visual confirmation of display bars and touch target movement is still pending; the serial `display test` and `touch test` commands return OK on hardware. The COM12 hardware pixel-capture path reconstructed the current PR #56 hierarchy from `51258ba` / Actions `29060900359` with matching firmware/host CRC `E72745BA`, a passing simulator diff, `public_rf_tx=false`, and `formats_sd=false`; a three-round all-tab health probe also passed cleanly. PR #51, PR #40, and PR #33 captures remain history for superseded layouts. Physical photos/manual review and final-release-SHA evidence are still pending.
- MeshCore Public group text TX/RX and signed advert TX/RX are implemented and validated through the serial console, firmware-local contacts can promote heard nodes by fingerprint, full advertised public keys are retained for DM targeting and have passed reboot persistence on `COM7`, contact favorite/mute flags persist across reboot, Public unread/read state persists across reboot, per-thread DM read cursors are implemented, firmware-local routes are learned from Public/advert path metadata, route detail diagnostics and a first route detail sheet are implemented, and serial DM flood TX plus the bounded DM store are validated. DM ACK/PATH receive parsing, direct-route TX selection, contact-row DM compose, first contact detail actions, a bounded Public History/search sheet, and a bounded scrollable DM thread/detail history sheet are implemented, and targeted outbound hardware DM receive by the local COM11 Meshcorebot has passed. `scripts/rf_full_acceptance_d1l.py` now provides a single COM12-side artifact path for identity, Meshcorebot status, outbound DM, controlled inbound DM, ACK/PATH, direct-route, health, and no-Public-command proof. Manual touch review, a passing full RF acceptance artifact, controlled inbound DM unread proof, and physical DM workflow review are still pending.
- Settings are implemented as an NVS-backed firmware model and have passed reboot persistence smoke on real D1L flash. The persisted radio profile is exposed through `settings get`, serial radio setters, and a touch Radio Settings sheet; the user-facing More page groups tools, connections, storage/maps, device, support, and advanced functions behind disclosure rows instead of a flat settings grid. PR #56 proves the collapsed hierarchy and all-tab navigation on COM12; expanded category touch behavior and the remaining system-page redesigns still need physical touch review.
- First-boot onboarding now persists a setup-complete flag, collects the node name, confirms Canada/USA defaults, keeps Wi-Fi/BLE/observer disabled, and generates/retains the local MeshCore identity. Optional-radio choices in onboarding remain limited until live Wi-Fi/BLE runtime support is enabled.
- `identity status` now generates and reports an NVS-retained Ed25519 local identity fingerprint. This is a minimal firmware identity model, not the final full MeshCore C++ store integration.
- MeshCore TX timestamps are retained monotonically in NVS to avoid duplicate filtering, but a real RTC/network time source is not wired yet. The current production patch keeps DM/Public TX from failing under onboard NVS capacity pressure by falling back to a monotonic RAM timestamp if only the timestamp persistence write fails; that fallback must be hardware-retested from the next Actions-built firmware package.
- MeshCore 3-byte companion framing is implemented and host-tested, but the live binary bridge to MeshCore protocol frames is not enabled in Phase 1.
- Packet log storage keeps a 128-row active RAM ring and persists the newest 8 rows through the compact NVS fallback. When the RP2040 SD file protocol is ready, packet history uses a split retained-store write path: SD receives the primary snapshot through coalesced flushes while NVS keeps the newest fallback rows, and a validated 4096-record segmented SD journal backs 100-row Packet tab pages with Load Older/Newer. Packet rows carry a bounded raw hex preview and expose serial filter/search/raw commands plus a touch terminal-style Packet tab with RX/TX/fail/error coloring, pause/resume, filter/search, route evidence, and normal/advanced packet detail; full hardware proof of 24h SD scrollback and deeper protocol decode are still pending.
- Optional SD-card data storage is now FAT32-only and non-destructive. Current ESP32 firmware reports safe `storage status`, explicit non-formatting `storage mount`, non-formatting `storage diag`, `storage setup` with `policy="no_device_format"`, and a touch SD Card sheet. The RP2040 bridge has no SD formatting command; users must provide FAT32 cards prepared on a computer. On mounted FAT32 media, DeskOS creates missing `/deskos` folders/manifests and can switch retained Public/DM message history, route history, and packet history to SD through `DESKOS_SD_FILE v=1` when the file-operation gate is ready. Serial storage canaries cover file writes, exports, sampled data export, synthetic map-tile cache commits, and a bounded allowed-provider HTTPS tile download command without Public RF or formatting. Touch Map Tiles setup can persist an allowed provider/attribution/zoom, but live touch downloads remain unavailable as `tile_render_pending` until tile rendering proof exists. Settings, identity, contacts, read-state, and boot-critical state remain onboard/fallback-backed. GPS/location-source integration, browser/geolocation integration, nearby GPS node pins, SD tile rendering/sideloading, and hardware proof remain pending.
- Older guarded-format hardware artifacts from July 1-2, 2026 are now obsolete for release readiness because the public policy forbids on-device formatting. Fresh SD acceptance evidence must use a FAT32 card prepared on a computer, prove auto-creation of the DeskOS directory structure, prove no-card NVS fallback, and prove non-FAT32/unmountable cards remain non-destructive with clear repair guidance.
- Current COM12/COM16 evidence from `1a29876` / GitHub Actions run `28714355561` proves the core SD path on the installed FAT32 card without Public RF or formatting: the official Seeed SD smoke passes on COM16, COM12 smoke reports `READY_SD`, raw RP2040 diagnostics complete, the file canary passes, correct/missing/existing-data boot scenarios pass, RP2040-unavailable fallback is captured after official smoke, retained Public/DM/route/packet history survives reboot, reboot/remount passes, the synthetic map-tile canary passes, and export canary/diagnostic export/sampled data export all pass. Public release is still blocked on the remaining physical SD evidence matrix: actual no-card and unformatted/non-FAT32 scenarios, FAT32-only rejection behavior for unusable media, <=32GB multi-card proof, no-format guidance evidence tied to unusable media, and power/electrical evidence.
- Phase 6 mesh visibility now exposes `signal`, `roomservers`, `repeaters`, `routes trace <fingerprint>`, and an opt-in DM-only `routes probe <fingerprint>` diagnostic plus Home/Packet-tab summary cards. A first touch Mesh Roles browser sheet and contact Route Trace sheet with `Ping` are implemented, and newest-build active-probe hardware proof passed in `artifacts/hardware/com12/routes_probe_a1afd4b.json` with no Public RF. Manual physical touch confirmation of opening/scrolling/closing these sheets and richer filters are still pending.
- Contact export emits MeshCore-compatible `meshcore://contact/add?...` payloads over serial with LVGL QR rendering enabled in the default build and requires a promoted contact with a retained 64-hex public key. The current host UI places Message and Contact options on Contact Detail, then Route, Export, Rename, favorite/mute state, and confirmed removal under Contact Options. Route, Export, Rename, Cancel, and Back all return without deleting; only the dedicated confirmation callback can remove a contact. The earlier QR-enabled build passed COM7 flash, standard smoke, targeted `contacts export <fingerprint>` proof, and active Public `test` RF regression, but the new hierarchy still needs an exact Actions-built COM12 capture, physical touch review, and scan/import proof with a MeshCore client.
- The Phase 3/4/5/6/D/E touch shell is implemented with first-boot onboarding, a five-destination Home/Messages/Network/Map/More hierarchy, a consolidated Device status card, Public `test` action, Public compose/history/search, DM compose/thread/detail flows, contact/detail/options/edit/export/route surfaces, offline Map/provider setup, Mesh Roles, Packets, nested More categories, connectivity/storage/radio/display/diagnostic sheets, toasts, and lock overlay. Message Detail, DM Thread, and the contact hierarchy use full-height nested pages; Contact Detail now keeps the common Message action separate from Contact Options, and removal is confirmation-only. PR #56 / source `51258ba` / Actions `29060900359` proves the current Home geometry on COM12 with CRC `E72745BA`; PR #58 / source `678bbcc` / Actions `29062818868` proves the full-height DM Thread geometry on COM12 with CRC `1214617A`, a passing simulator diff, a clean scroll probe, and a clean three-round all-tab probe; PR #35 / Actions `28727064923` proves all 12 keyboard callers. Navigation, shared chrome, the Home and top-level More renderers with stable enum actions, keyboard policy, an explicit canonical content-root renderer contract, and single-active modal ownership are split, and every modal hides background dock navigation. #16/#6 still need owned full-height Network/routes, Map, Packets, contact, and remaining settings page domains; UI-task command ownership; remaining sheet-specific keyboard submit/cancel ownership; and stronger callback-lifetime/single-root contracts. The contact hierarchy has host implementation and contract coverage, but not yet issue-matched COM12 pixel/touch evidence. Nearby GPS node pins, GPS/location-source integration, actual tile rendering/sideload browsing, BLE companion transport, Display runtime persistence, OTA, and manual physical review of the newest touch surfaces are not complete yet.
