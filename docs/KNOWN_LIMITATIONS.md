- The first #69 durability slice removes synchronous full-route rewrites and storage reinitialization from hot RX observations. A dedicated persistence worker gives the first material observation a full 5-second coalescing floor, flushes nonmaterial-only dirty state by 30 seconds, and handles explicit migration/reconciliation immediately; failed forced flushes now honor the same five-second retry floor instead of hammering storage. Controlled reboot force-flushes or cancels, and `routes` exposes commit/coalesced/failure/dirty/reconcile truth. SD retains the newest 16-route primary snapshot while NVS mirrors only the newest 4 routes. Retained Public/DM/route/packet mirrors use a dedicated 124 KiB `d1l_retained` NVS partition, marker format v2 with two slots in the separate 4 KiB metadata sector, a committed anchor inside the dedicated NVS, and a completion sentinel in default NVS rather than competing with settings, identity, Wi-Fi, contacts, and nodes in the 24 KiB default partition. Upgrade reads prefer the dedicated partition, copy an old default-NVS retained key before erasing only that key, preserve and fail closed on divergent downgrade/re-upgrade copies, and commit the completion sentinel only after migration succeeds. Marker- or sentinel-owned recovery never explicitly erases `d1l_retained`, sentinel-only marker repair requires the already-committed anchor, and the sole blank owned-state resume is marker 1 valid, marker 2 erased, and no completion sentinel. Published pre-anchor v1 markers remain valid ownership: firmware initializes and commits the anchor without a data erase, migrates, commits the sentinel, then rewrites both marker slots as v2. `storage status.retained_nvs` must report `marker_ready=true`, `markers_complete=true`, `anchor_ready=true`, `sentinel_ready=true`, and `external_init_required=false` before the backend is release-ready. A late or transiently unreadable SD primary is read and merged by epoch/total before any overwrite, including chronologically replayed live overlays; persistent SD read failure still commits an independently snapshotted compact NVS fallback while keeping SD reconcile and controlled reboot fail-closed. A synchronized monotonic backend generation forces this read/merge again after every observed remove/reinsert cycle even when the worker never samples the intermediate disabled level. Every retained SD writer captures that generation and rechecks it immediately before replace-rename; a detected mid-write media edge leaves the primary untouched and does not issue a cleanup delete against the replacement card. Persisted route envelopes reject unterminated/non-printable strings, invalid sequence metadata, duplicate sequences, and impossible path fields before exposing rows. A same-epoch newer legacy snapshot wins for rollback freshness while a newer v2 epoch blocks stale legacy resurrection. Volatile UI route canaries use a separate preview slot and do not consume durable sequence/count/revision/capacity, and a failed `routes clear` blocks every periodic/forced persistence attempt until an explicit clear succeeds. The current source now gives Public/DM/packet route-equivalent late-card reconciliation and independently retryable NVS targets. Public/DM schema v4 persists a 64-bit clear lineage and treats valid device-local NVS as authoritative over a foreign removable lineage; packet journal retries use validated sequence-derived fixed offsets and the canonical RP2040 zero-byte EOF reply. - ESP-IDF's `nvs_flash_init_partition` is not a read-only discriminator: initialization may erase or activate a corrupt page. Firmware therefore performs zero retained-region erases and does not call NVS initialization on ambiguous nonblank bytes with neither marker nor sentinel ownership; it reports `external_init_required=true`. A separately audited installer/hardware step must verify the supported predecessor layout before erasing only `d1l_retained`; firmware then verifies blank first use and establishes marker 1, NVS, anchor, marker 2, migration, and sentinel in order. The known failed pre-anchor recovery cannot be authorized by a similar-looking live status: the installer also requires the hash-pinned incident manifest, exact historical receipt hashes, exact table artifact, live ESP32-S3 MAC, the audited prior MeshCore identity, and exact full raw-region SHA, then records durable pre-erase/erase/verify stages without retaining a raw backup. Simultaneous loss of both markers and the default sentinel, including an anchor-only NVS, stays fail-closed and preserved until that external decision; it is not an automatic deletion case.
- NVS initialization is fail-closed and non-destructive: DeskOS no longer erases the whole NVS partition automatically for `ESP_ERR_NVS_NO_FREE_PAGES` or an unsupported stored NVS version. The boot event and `health` expose a per-boot nonce plus `nvs_ready`/`nvs_error`; persisted settings remain untouched for explicit recovery. In this degraded mode NVS-backed features may be unavailable until the underlying capacity/version problem is resolved, and exact hardware recovery UX remains open under #69.
