# Known Limitations

As of the 2026-07-01 D1L COM12 production-roadmap checkpoint:

- Manual visual confirmation of display bars and touch target movement is still pending; the serial `display test` and `touch test` commands return OK on hardware.
- MeshCore Public group text TX/RX and signed advert TX/RX are implemented and validated through the serial console, firmware-local contacts can promote heard nodes by fingerprint, full advertised public keys are retained for DM targeting and have passed reboot persistence on `COM7`, contact favorite/mute flags persist across reboot, Public unread/read state persists across reboot, per-thread DM read cursors are implemented, firmware-local routes are learned from Public/advert path metadata, route detail diagnostics and a first route detail sheet are implemented, and serial DM flood TX plus the bounded DM store are validated. DM ACK/PATH receive parsing, direct-route TX selection, contact-row DM compose, first contact detail actions, a bounded Public History/search sheet, and a bounded scrollable DM thread/detail history sheet are implemented, and targeted outbound hardware DM receive by the local COM11 Meshcorebot has passed. `scripts/rf_full_acceptance_d1l.py` now provides a single COM12-side artifact path for identity, Meshcorebot status, outbound DM, controlled inbound DM, ACK/PATH, direct-route, health, and no-Public-command proof. Manual touch review, a passing full RF acceptance artifact, controlled inbound DM unread proof, and physical DM workflow review are still pending.
- Settings are implemented as an NVS-backed firmware model and have passed reboot persistence smoke on real D1L flash. The persisted radio profile is now exposed through `settings get`, serial radio setters, and a touch Radio Settings sheet; Settings now uses friendly groups for Wireless, MeshCore, Storage, Display, Diagnostics, and About, with staged Display controls and a Diagnostics sheet for health/crashlog/export/soak entry points. The radio editor slice has passed hardware flash, standard smoke, targeted serial profile proof, and active Public `test` RF regression on `COM7`; the grouped Settings slice is host/simulator covered and awaits Actions-built COM12 hardware flash. Manual touch review is still pending.
- First-boot onboarding now persists a setup-complete flag, collects the node name, confirms Canada/USA defaults, keeps Wi-Fi/BLE/observer disabled, and generates/retains the local MeshCore identity. Optional-radio choices in onboarding remain limited until live Wi-Fi/BLE runtime support is enabled.
- `identity status` now generates and reports an NVS-retained Ed25519 local identity fingerprint. This is a minimal firmware identity model, not the final full MeshCore C++ store integration.
- MeshCore TX timestamps are retained monotonically in NVS to avoid duplicate filtering, but a real RTC/network time source is not wired yet. The current production patch keeps DM/Public TX from failing under onboard NVS capacity pressure by falling back to a monotonic RAM timestamp if only the timestamp persistence write fails; that fallback must be hardware-retested from the next Actions-built firmware package.
- MeshCore 3-byte companion framing is implemented and host-tested, but the live binary bridge to MeshCore protocol frames is not enabled in Phase 1.
- Packet log storage keeps a 128-row active RAM ring and persists the newest 8 rows through the compact NVS fallback. When the RP2040 SD file protocol is ready, packet history uses a split retained-store write path: SD receives the primary snapshot through coalesced flushes while NVS keeps the newest fallback rows. Packet rows carry a bounded raw hex preview and expose serial filter/search/raw commands plus a touch terminal-style Packet tab with RX/TX/fail/error coloring, pause/resume, filter/search, route evidence, and normal/advanced packet detail; full hardware proof of long SD scrollback and deeper protocol decode are still pending.
- Optional SD-card data storage is now an important roadmap item with ESP32-side RP2040 status/setup plumbing and a CI-buildable RP2040 SD bridge target. Current ESP32 firmware reports safe `storage status`, explicit non-formatting `storage mount`, non-formatting `storage diag`, `storage setup`, and a touch Storage Setup sheet, can switch retained Public/DM message history, route history, and packet history to SD through `DESKOS_SD_FILE v=1` when the bridge is ready, and does not run boot-time or incidental formatting. Serial-only storage canaries cover file writes, exports, sampled data export, and synthetic map-tile cache commits without Public RF or formatting. The operator has allowed formatting the SD card inserted in the D1L for production validation, but only through the guarded explicit confirmation path on an unformatted or sacrificial test card. Settings, identity, contacts, read-state, and boot-critical state remain onboard/fallback-backed. Live network map tile downloads, GPS/location-source integration, browser/geolocation integration, nearby GPS node pins, and SD tile rendering/sideloading remain pending.
- SD bridge hardware update 2026-07-01: Actions run `28502609315` for commit `b8d374246310ac761cbd965ae59a0bd0e1b6883f` flashed the ESP32 on COM12 after the RP2040 raw-probe UF2 from run `28502095977`. COM12 preflight `artifacts/hardware/com12/rp2040_sd_preflight_format_reporting_28502609315.json` proves `rp2040 ping` works, safe `storage status` reports `state="mount_required"` without touching SD, explicit `storage mount` is responsive, and the bounded raw probe detects the inserted card as `setup_required`/`format_supported=true` without Public RF or formatting. Guarded format evidence `artifacts/hardware/com12/sd_format_guarded_reporting_28502609315.json` now reports the non-ready format result as `ok=false` instead of `format_performed=true`, and `artifacts/hardware/com12/sd_file_canary_blocked_setup_required_28502609315.json` confirms SD writes stay blocked while the file-operation gate is not ready.
- SD guarded-format update 2026-07-01: `artifacts/hardware/com12/sd_boot_prepare_unformatted_736ccfc.json` exercised the operator-approved unformatted-card confirmation path and stayed safe (`public_rf_tx=false`, `formats_sd=false`, `format_allowed=true`, health ready), but failed readiness as `classification="format_confirmed_not_ready"` because the RP2040 bridge timed out before returning a confirmed format result. Actions run `28553272721` for commit `736ccfc` produced the verified RP2040 UF2 (`ACBA21708C9DAB5CEE19305F9BC31947357C71D5298FAB07AC64DB5296E089D3`), but no mounted UF2 bootloader volume was available, so the bridge still requires physical UF2/BOOTSEL action for copying. `artifacts/hardware/com12/rp2040_preflight_736ccfc_after_format_timeout_settle.json` proves the COM12 RP2040 UART/protocol/diag paths respond and the inserted card reaches `setup_required` with raw-card-present/mount-failed diagnostics, but `ready_for_sd_acceptance=false`. The follow-up patch keeps SdFat formatter progress as a UART keepalive, adds a newline fence before the terminal `DESKOS_SD_FORMAT` reply, extends the ESP32 confirmed-format wait to 660 seconds, and records `format_command_sent` separately from `format_confirmed`; full SD auto-prepare, retained history, map tile, export, and reboot/remount proof remain blocked until the next Actions-built ESP32 image and RP2040 bridge are flashed, the guarded format returns ready, and the SD matrix is rerun.
- Phase 6 mesh visibility now exposes read-only `signal`, `roomservers`, `repeaters`, and `routes trace <fingerprint>` diagnostics plus Home/Packet-tab summary cards, and the serial/summary slice is built, flashed, smoke-tested, reboot-checked, and Public `test` RF-regression tested on `COM7`. A first touch Mesh Roles browser sheet and read-only contact Route Trace sheet are also implemented; manual physical touch confirmation of opening/scrolling/closing them is still pending. Active RF ping/trace helpers and richer filters are still pending.
- Contact export now emits MeshCore-compatible `meshcore://contact/add?...` payloads over serial and adds a Contact Detail `Export` action with LVGL QR rendering enabled in the default build. It requires a promoted contact with a retained 64-hex public key. The QR-enabled build passed COM7 flash, standard smoke, targeted `contacts export <fingerprint>` proof, and active Public `test` RF regression; manual touch open/close review and scan/import proof with a MeshCore client are still pending.
- The Phase 3/4/5/6 touch shell is implemented with first-boot onboarding, Home, Public `test` action, first Public free-text composer, bounded Public History/search, first contact-row DM composer, first contact detail sheet with DM/favorite/mute/Trace/Edit actions, a contact Edit sheet for alias rename and promoted-contact Forget, bounded DM thread/detail retained history with `Reply` and per-thread `Read`, persisted recent Public/DM preview rows, Public/DM unread counters, newest contact/heard-node rows, Home Wi-Fi/BLE/SD setup chip routing, an offline Map tab with SD tile-cache policy/readiness, retained route evidence, serial-configured manual center display, and first-open touch D1L pin picker/persistence, first Mesh Roles browser sheet, first contact Route Trace sheet, Packet-tab route rows with a first route detail sheet, terminal-style Packet-tab rows with filters/search/pause and normal/advanced packet detail, grouped Settings with Wireless/MeshCore/Storage/Display/Diagnostics/About, a Radio Settings sheet, Storage Setup sheet, staged Wi-Fi/BLE setup sheets, staged Display sheet, Diagnostics sheet, advert sheet, toasts, and lock overlay, but active route ping/trace helpers, nearby GPS node pins, GPS/location-source integration, live map tile downloads, BLE companion transport, live Wi-Fi scan/connect, Display runtime persistence, and OTA are not implemented yet.
- `tools/ui_simulator.py` now provides deterministic 480x480 host screenshots, schema-v2 required-label/text-overflow checks, a touch-target map with expanded 44x44 boxes, expected flow coverage for onboarding, lock/unlock, Public compose/history/search, DM thread read/reply, contact detail/edit/export/trace, Map policy/location picker/manual center, packet search/detail/routes, Mesh Roles, Radio, Storage, Display, Diagnostics, and Advert, plus RF/destructive/no-format action flags. - The Public composer, bounded Public History/Search sheets, contact-row DM composer, contact detail sheet, contact Route Trace sheet, contact export sheet, Radio Settings sheet, bounded retained-history DM thread sheet, per-thread DM read action, Messages unread/read action, route detail sheet, and packet detail sheet have passed host contract/simulator tests.
